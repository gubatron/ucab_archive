Proyecto 2


Introducción

Los sistemas distribuidos presentan características muy relevantes y diferenciables de cualquier otro tipo de sistema. La transparencia de localidad permite independizar la relacion física que se establece entre piezas de software cliente y servidor, la tolerancia a fallas ofrece sistemas altamente disponibles y la sincronización distribuída permite (entre otras cosas) tomar decisiones en función a información que puede estar particionada o replicada entre varias maquinas y/o servidores.


Descripción General

Durante este segundo proyecto se persigue que el estudiante diseñe e implemente un sistema distribuido que cumpla con las concepciones básicas indicadas en la INTRODUCCION.

Se pretende crear un pequeño manejador de tuplas distribuidas. A dicho manejador de ahora en adelante lo denominaremos TUPLAS.

TUPLAS estará formado por dos piezas de software diferenciadas. La primera es el "engine" como tal, se trata de una serie de servidores (demonios UNIX llamados TuplaD) que se encontrarán distribuidos en la red y sobre los cuales residirán los diferentes conjuntos de tuplas. La segunda pieza es la herramienta administrativa y de manipulación de datos (Tupla-Admin), se trata de un cliente que sabe interactuar con el servicio distribuido TUPLAS.

Concepto de tuplas

La tupla es una secuencia ordenada de datos cuyo primer elemento es la clave única de la tupla. Los elementos que componen una tupla se tratan como caracter pues no existe la nocion de tipo de dato. La estructura de una tupla ti es:

ti = (d0,d1,...,dn) (dimensión n+1)

Cada tupla tiene asociada una dimensión que representa el número de elementos, (incluyendo la clave)


Donde d0 es la clave primaria única y cada di (i:1..n) es un string.

En los servidores TuplaD residirán conjuntos de tuplas. Un conjunto de tuplas se puede definir como:

T = {t1,.....,tn}

Donde ti es una tupla con una clave única en el sistema.

Funcionamiento General

El servicio TUPLAS consta de una serie de servidores (demonios UNIX: TuplaD) que se montan en diferentes máquinas y se comunican entre ellos vía Sockets y con el resto del mundo vía RMI. Estos servidores son capaces de manipular conjuntos de tuplas que son definidos y manipulados por el usuario a través de Tupla-Admin.

Cuando se define un conjuntos de tuplas se puede indicar que el mismo residirá replicado, particionado o segmentado entre diferentes servidores del sistema. En cualquiera de estos casos el sistema debe garantizar que el acceso a los datos por parte del usuario será transparente y consistente, por tanto deben considerarse todos los casos asociados a sincronización y ejecución de transacciones distribuidas (por ejemplo, exclusión mutua).

  • Se dice que un conjunto de tuplas está replicado si existe al menos 1 copia en otro servidor. Las operaciones de modificación de la información se deben ejecutar de manera transaccional en todos los servidores que contengan una réplica de la réplica.

    Ejemplo: Dado T ={t1,....,tn} , S1 y S2 servidores TuplaD donde reside T se tiene que:

    S1.T = S2.T

    Se debe implementar un algoritmo de two phase commit para garantizar integridad en el caso de operaciones de modificación de datos sobre las réplicas.

  • Un conjunto de tuplas está particionado si la unión de las tuplas que se encuentra en varios servidores forma el conjunto de tuplas final.

    Ejemplo: Dado T = {t1,....,tn} , S1 y S2 servidores TuplaD donde reside T se tiene que:

    (S1.T intersección S2.T) = vacío y (S1.T' U S2.T'' = T)

  • Finalmente, un conjunto de tuplas está segmentado si en dos o más servidores se encuentran las tuplas divididas como lo muestra el ejemplo.

    Ejemplo: Dado T = {t1,....,tn} , S1 y S2 servidores TuplaD donde reside T se tiene que:

    S1.T' = {(d0, d1,...,dn/2), (e0,e1,...,en/2)} y S2.T'' = {(d0,dn/2+1,...,dn), (e0, en/2+1,...,en)}. Notar que el primer elemento de las subtuplas es la clave.

Finalmente, se pretende que el sistema sea capaz de detectar si uno de sus integrantes se cae para de esa manera poder indicarle al usuario cuando los datos no se encuentran disponibles por falla del servicio o por ausencia de los mismos.


Operaciones sobre Tuplas

Un conjunto de tuplas ofrece los siguientes métodos para su manipulación:

  • Crear (Nombre, Dimensión, Tipo, Lista de Servidores) = Status

    Donde = Nombre: Identificador del conjunto de tuplas
    Dimensión: Número de elementos de una tupla (debe ser >= 2)
    Tipo: Indica si es segmentado, replicado o particionado
    Lista de Servidores: Nombre de las máquinas donde se desea que resida el conjunto de tuplas.

    Las operaciones sobre las tuplas pueden manejar un esquema de locking centralizado para proveer la exclusión mutua.

  • Eliminar (Nombre) = Status

  • Insertar (Nombre, ti) = Status

  • Borrar (Nombre, Clave) = Status

  • Buscar (Nombre, Clave) = Lista de Valores (tupla)

  • Actualizar (Nombre, Clave, Posicion, Valor) = Status

  • Configuración (Nombre) = Información de configuración del conjunto de tuplas


Servidores Tuplad

Se trata de los componentes del sistema que manejan todo lo referente a los conjuntos de tuplas así como la interacción del cliente con dicha TUPLAS.

Son demonios UNIX construídos en JAVA que interactúan entre si a través de sockets. Son los encargados de garantizar que el sistema se comporta de manera distribuida, para llevar esto a cabo, tienen definido un protocolo de comunicación.

Ofrecen al usuario la posibilidad de ejecutar cualquier tipo de operación sobre un conjunto de tuplas, mediante Tupla-Admin, la interacción que se establece con dichos usuarios es a través de RMI Multicast.

Los servidores deberán balancear la carga al momento de crear los conjuntos, es decir, repartir los conjuntos uniformemente en los servidores.


Herramienta de Manipulación

Es el software cliente que debe ofrecer la posibilidad de configurar un ambiente TUPLAS ya montado, asi como la manipulación de los conjuntos de tuplas.

Deberá proveer una interfaz que permite visualizar el estado del sistema completo.
